## 文档一：DeepAgents 回顾报告

**文件**：`deepagents-retrospective.md`

### 标题
DeepAgents Retrospective — DeepAgents 架构全面回顾

### 历史背景
DeepAgents 是一个基于 LangGraph 构建的多 Agent CLI 框架，核心创新是 **Task Tool Delegation Pattern**（主 Agent 通过 `task(description, subagent_type)` 将工作委派给子 Agent）。本回顾报告在项目达到一定成熟度后编写，目的是系统性地审视现有架构的优势、问题和改进方向。

### 核心内容

1. **项目背景与核心创新**
   - 2025 年初，多 Agent 系统的主流模式是编排型（orchestrator）和群体型（swarm）
   - DeepAgents 采用"委派"模式：主 Agent 专注于理解用户意图和任务分解，子 Agent 作为"员工"执行具体工作
   - 实现了 `SubAgentMiddleware`、`SubAgentManager`、`TaskToolGenerator` 等核心组件

2. **架构亮点**
   - **SubAgentMiddleware**：统一的中间件，拦截 task tool call，根据类型分发到不同子 Agent
   - **TaskToolGenerator**：从 `subagent.yaml` 声明文件自动生成 tool 定义
   - **SubAgentManager**：支持并发管理多个 subagent 实例
   - **事件系统**：通过回调函数实现跨 Agent 的通知（如代码提交后通知 code-review）

3. **已识别的问题**
   - **问题 1：LLM 上下文窗口是核心瓶颈**。主 Agent 的 `max_tokens` 只有 5120，在复杂任务中容易截断
   - **问题 2：LLM 串行调度导致并发度低**。LangGraph 的 tool executor 逐个串行执行子任务
   - **问题 3：配置分散**。SubAgent 配置在多个地方（`subagent.yaml`、`agent.yaml`、环境变量）
   - **问题 4：Python 的 GIL 限制了真正的并发**

4. **提出的改进方向**
   - 短期：优化上下文管理（消息裁剪、摘要）
   - 中期：引入主动调度器（Proactive Scheduler），解耦提交与执行
   - 长期：考虑用 Go 重写以获得更好的并发能力和性能

5. **经验教训**
   - "最好的 Agent 系统是让 LLM 做最少决策的系统"
   - 并发和上下文管理是多 Agent 系统的两个核心挑战
   - 声明式配置优于硬编码

### 关键结论
- DeepAgents 的委派模式验证了"主 Agent 做决策、子 Agent 做执行"的有效性
- LangGraph 提供了良好的图编排能力，但在高并发场景下受限
- Python 的 GIL 和生态限制是推动 Go 重写的主要动因
- 上下文管理（而非模型能力）是多 Agent 系统的真正瓶颈

## 文档二：RFC — Go 重写提案

**文件**：`rfc-go-rewrite.md`

### 标题
RFC: Rewrite deepagents in Go with Google ADK — 用 Google Agent Development Kit (Go) 重写 DeepAgents

### 历史背景
在 DeepAgents 回顾报告之后，团队评估了多个技术路线。Google 的 ADK（Agent Development Kit）提供了 Go 语言实现（`adk-go`），其基于 Runner 的多 Agent 架构与 DeepAgents 的委派模式高度契合。此 RFC 提出用 ADK Go 重写整个系统，以解决 Python 版本的并发和性能瓶颈。

### 核心内容

1. **动机**
   - Python GIL 限制了真正的并发
   - ADK Go 的 Runner 架构原生支持多 Agent 并发
   - 部署简化：Go 编译为单一二进制，无需 Python 虚拟环境和依赖管理
   - Google 官方维护的框架，长期支持有保障

2. **技术对比：Python 版 vs Go 版**
   - 上下文管理：Python 版手动管理 messages 列表；Go 版 ADK Runner 自动按 Agent 隔离上下文
   - 并发：Python 版受 GIL 限制；Go 版原生 goroutine 并发
   - 部署：Python 版需要 venv + pip；Go 版单一二进制
   - 工具生态：Python 版用 LangGraph；Go 版用 ADK Runner + Custom Agent

3. **架构设计**
   - 使用 ADK Runner 管理 Agent 生命周期和路由
   - Custom Agent（`GeneralPurposeAgent` / `CodeAgent` / `SearchAgent`）实现具体业务逻辑
   - `SubAgentManager` 管理子 Agent 的创建和配置
   - 保留与 Python 版的接口兼容性（CLI 命令、配置格式等）

4. **迁移策略**
   - **Phase 0**：验证和基础设施搭建（adk-go 调研、项目结构、CI）
   - **Phase 1**：核心 Agent 功能（3 个 subagent + runner）
   - **Phase 2**：文件系统工具（read_file、write_file、edit_file、run_shell 等）
   - **Phase 3**：会话持久化（SQLite 实现 SessionService）
   - **Phase 4**：MCP 协议集成（`go-mcp`）
   - **Phase 5**：产品化（CLI、Telemetry、Human-in-the-loop）

5. **风险评估**
   - **adk-go API 不成熟**：当前 v0.2.0+，API 可能频繁变化
   - **迁移成本**：需要重新实现所有工具和 Agent 逻辑
   - **生态差异**：ADK 的设计哲学与 LangGraph 不同，需要学习适应
   - **非 Gemini LLM 接入**：ADK 优化为 Gemini，其他 LLM 需要额外适配

### 关键结论
- Go + ADK 方案在并发、部署、长期维护方面显著优于 Python 版
- 建议分 5 个阶段逐步迁移，降低风险
- 最大风险在于 adk-go 的 API 成熟度和非 Gemini LLM 的适配
- 开放问题包括状态持久化方案选择、流式输出处理、CLI 框架选择等

## 文档三：RFC — 主动调度器与事件驱动 Agent 激活

**文件**：`rfc-proactive-scheduler-and-event-system.md`

### 标题
RFC: Proactive Scheduler & Event-Driven Agent Activation — 主动调度器与事件驱动的 Agent 激活机制

### 历史背景
DeepAgents 回顾报告中识别出"LLM 串行调度导致并发度低"和"调度依赖 LLM 决策"两大核心问题。此 RFC 是在 Python 版本框架下提出的改进方案，旨在不重写整个系统的情况下，通过引入调度中间件和事件总线来提升并发能力和自动化水平。

### 核心内容

1. **问题陈述**
   - **问题 1：并发度低** — 即使 LLM 在一个 turn 中发出多个 task tool call，LangGraph 仍逐个串行执行
   - **问题 2：调度依赖 LLM 决策** — 每个子任务的触发都需要主 Agent 的 LLM 推理，导致：
     - 每轮协作都需要一次 LLM 调用（成本 + 延迟）
     - LLM 无法预见任务执行中产生的衍生需求
     - 复杂多步协作需要多轮 LLM turn

2. **现有并发机制及其不足**

   | 机制 | 并发能力 | 不足 |
   |---|---|---|
   | 同步 SubAgent (`task` tool) | 串行阻塞 | 一次只有一个 subagent 在工作 |
   | 异步 SubAgent (`start_async_task`) | 真并发 | 依赖远程 LangGraph Server，无法本地使用 |
   | Template Fan-Out (`Send`) | LangGraph 原生并行 | 仅在 template 路由层生效，需预先分解 |

3. **架构设计**
   - **SchedulerMiddleware**：统一的调度中间件，包含三个核心组件：
     - `TaskQueue`：优先级 + 依赖感知的任务队列
     - `Scheduler`：调度循环，从队列取任务分发到 WorkerPool
     - `WorkerPool`：基于 `asyncio.Semaphore` 的并发执行池
   - **EventBus**：事件发布与规则匹配系统
     - 定义 `AgentEvent`（event_type、source、payload）
     - `EventHandler`（rule + handler）实现事件到任务的自动转换
     - 支持 cron 表达式和事件触发两种方式

4. **核心数据结构**
   - **TaskRecord**：包含 task_id、description、subagent_type、priority、dependencies、status 等字段
   - **TaskPriority**：CRITICAL(0) → HIGH(1) → NORMAL(2) → LOW(3) → BACKGROUND(4)
   - **状态流转**：pending → ready → running → success/error/cancelled

5. **LLM 接口工具**
   - `submit_task`：提交单个任务，立即返回 task_id，不阻塞
   - `submit_task_group`：批量提交一组任务，支持组内依赖声明
   - `await_tasks`：等待一组任务完成，等待期间所有任务并行推进

6. **事件驱动协作示例**
   - 事件规则："task_completed + subagent_type=code-review → 自动创建 fix-issues 任务"
   - Cron 规则：每 30 分钟自动执行一次代码健康检查
   - 事件与任务通过 EventBus 自动关联，无需 LLM 介入

### 关键结论
- Proactive Scheduler 将"提交即阻塞"改为"提交-调度-执行-收集"模型，显著提升并发能力
- EventBus 实现了 Agent 间的自动协作，减少了对 LLM 决策的依赖
- 该方案是 Python 版本下的渐进式改进，与 Go 重写路线互斥（二选一）
- 核心创新点：任务依赖图、优先级调度、事件自动触发、cron 定时任务

## 整体总结

这三份归档文档记录了 DeepAgents 项目从 Python/LangGraph 架构向技术路线抉择演进的关键思考：

1. **架构验证**：DeepAgents 的委派模式（主 Agent 决策 + 子 Agent 执行）被证明是有效的多 Agent 协作方式
2. **瓶颈识别**：Python GIL、LLM 串行调度、上下文窗口限制是三大核心瓶颈
3. **路线分歧**：团队提出了两条改进路线 — Python 渐进式改进 vs Go 彻底重写
4. **设计遗产**：无论是哪条路线，任务优先级调度、事件驱动协作、声明式配置等设计理念都将保留
